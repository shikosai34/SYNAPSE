# 認証・権限ガイド (「認証が必要です」と出たときに読む)

このドキュメントは、操作中に **「認証が必要です」「権限がありません」** などのメッセージが出たときに、
**何をすればよいか**をまとめたものです。あわせて、なぜ認証が必要なのか(特に画像アップロード)の
背景も記載します。

---

## 1. まず結論: メッセージ別の対処

| 出るメッセージ (HTTPステータス) | 意味 | 対処 |
| --- | --- | --- |
| `認証が必要です` / `認証されていません` (401) | ログインセッションが無い/切れている | 画面右上などからログインし直す(better-auth のメール+パスワード)。下記「2. ログインの種類」参照 |
| `アップロード権限がありません` (403) | ログインはしているが、どのサークル/イベントにも**所属していない** | いずれかのサークル/イベントに**メンバーとして追加**してもらう。招待を受けて `membership` を持つ必要あり |
| `管理者権限が必要です` (403) | その操作にはオーナー/管理者ロールが必要 | サークルなら `circle_manager`(オーナー)、イベントなら `event_manager` を持つ人に依頼、または権限を付与してもらう |
| `権限がありません` (403) | 対象のサークル/イベントに対する操作権限が無い | 自分が所属している正しいサークル/イベントに**スペース切替**してから操作する |
| ログイン失敗が続いて弾かれる (429, `Retry-After`) | レート制限/アカウントロックアウト(5回失敗で15分ロック) | 表示された時間だけ待つ。急ぐ場合はシステム管理のロック解除で解除可能 |

> ほとんどの 401 は「ログインし直す」、ほとんどの 403 は「所属/ロールが足りない or スペースが違う」です。

---

## 2. ログインの種類 (この製品には複数ある)

FesFlow には歴史的経緯で **複数の認証方式が併存**しています(認証現状メモ の再設計対象)。
自分がどれで詰まっているかを見分けてください。

1. **アカウントログイン (better-auth / メール+パスワード)**
   - 管理コンソール(サークル/イベント/システム)にログインするための土台。
   - `認証が必要です`(401)はほぼこれ。サインイン画面からメール+パスワードでログイン。
   - メール送信は未設定のため、**確認メール/パスワードリセットメールは飛びません**。パスワードを忘れた場合は
     super_admin(`sato.t.5970@gmail.com`)に相談。

2. **サークルPINログイン** (`/api/memberships/authenticate-pin`)
   - レジ端末などでサークルに入るための PIN。email 必須。5回失敗でロックアウト。

3. **イベントログイン** (`/api/festivals/login`)
   - イベント名+サークル名+パスワードでの照合。失敗理由は列挙防止のため一律同一メッセージ。

4. **来場者(パスワードレス)**
   - リストバンドの `eventUser.id` をベアラーとして端末に保持。ログイン画面は無い。
   - レジで「顧客未スキャン」になるのはこれ(＝お客様のQR/リストバンド未スキャン)。

### スペース(所属)切替で直る 403
複数のサークル/イベントに所属していると、**いま選択中のスペース**に対してしか操作できません。
`権限がありません`(403)が出たら、ヘッダーのスペース切替で**対象のサークル/イベントに切り替えて**から
やり直してください。

## 3. なぜ画像アップロードは認証が必要になったのか

> 質問:「画像アップロードはなんで認証必要になったんだっけ?」

**2026-07-06 のセキュリティ監査 (項目 M4「アップロード濫用対策」) で認証必須化しました。**
実装は `apps/api/src/index.ts` の `POST /api/upload`。

### 経緯
- それ以前はアップロードが**無認証**で叩けました。
- FesFlow は来場者が**自己登録(パスワードレス)**できるため、「ログインだけを条件」にしても
  **実質誰でも**10MB のファイルを**無制限に**アップロードできてしまいます(ストレージ濫用・不正コンテンツの温床)。
- そこで、アップロード導線は本来 **イベント/サークル/メニュー管理者の操作**に限られることから、
  次の2段構えを要件にしました。

### 現在の要件 (2つとも満たす必要がある)
1. **better-auth のログインセッションがあること** → 無ければ `401 認証が必要です`
2. **有効な `membership` を1つ以上持っていること**(スタッフ以上の所属) → 無ければ `403 アップロード権限がありません`

あわせて、拡張子ホワイトリスト(画像 jpg/png/webp・フォント ttf/otf/woff/woff2)と
**10MB のサイズ上限**も課しています。

### つまり画像アップロードで弾かれたら
- `401` → ログインし直す。
- `403` → どのサークル/イベントにも所属していない状態。招待を受けてメンバーになる必要がある。
  (新規登録直後で所属ゼロのアカウントはこの状態になります)

## 4. 開発者向けメモ

- 認証・権限まわりの現状スナップショットは `docs/` 外のメモリ 認証現状メモ にも要約あり。
- 401/403 を返す主な箇所:
  - 画像アップロード: `apps/api/src/index.ts` (`POST /api/upload`)
  - アカウント: `apps/api/src/routes/account.ts`
  - サークル/メニュー/注文: `apps/api/src/routes/{circle,menu,order}.ts`
  - イベント: `apps/api/src/routes/event.ts`
  - レート制限/ロックアウト: `apps/api/src/utils/rate-limit.ts`
- 認証方式は将来的に一本化を検討中(better-auth 中心 or 独自 PIN の整理)。
