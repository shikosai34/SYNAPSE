## エグゼクティブサマリ

現状は「単一学園祭の内輪運用」としては直近の修正でかなり堅くなっているが、**不特定多数のテナントが自分でサインアップして使う SaaS** として公開するには、以下の観点でまだ穴がある。

- **最上位管理者への昇格経路が、メール検証なしのサインアップと結びついている**（Critical）。
- **パスワード / PIN のハッシュ強度が極端に低い**（bcrypt cost=4）。DB 漏洩時にほぼ即解読される（High）。
- **テナント分離が甘い**。イベント / サークル / メニュー一覧が無認可で横断参照でき、SaaS のマルチテナント境界になっていない（High）。
- **リストバンドのベアラーモデル**が、個人情報（誕生日・ニックネーム）の閲覧 / 上書きと、なりすましスタンプ / 抽選口数稼ぎに悪用できる（High〜Medium）。
- **無認可の自動レコード生成**（`eventUser` 等）による DB 膨張 / コスト増 / DoS（Medium）。
- SaaS に必須の周辺実装（メール検証、パスワードリセット、監査ログ、テスト、バックアップ、決済・利用規約・プライバシー）が未整備（High〜Medium）。

## Critical

### C-1. `INITIAL_SUPER_ADMIN_EMAIL` × メール検証なしサインアップ = 全システム乗っ取り
- 場所: `packages/auth/src/index.ts`（`emailAndPassword: { enabled: true }` のみ、`requireEmailVerification` 等なし）／ `apps/api/src/utils/auth.ts` `getAdminSession()` の自動昇格ロジック（26〜52行）／ `apps/api/wrangler.jsonc`（42行に `INITIAL_SUPER_ADMIN_EMAIL: "sato.t.5970@gmail.com"` が**平文でコミット**）。
- 問題:
  1. better-auth の email+password が**メール検証なし**で有効。誰でも任意メールで即ログイン可能なセッションを得られる。
  2. `getAdminSession` は「セッションのメール文字列が `INITIAL_SUPER_ADMIN_EMAIL` と一致」しただけで `super_admin` メンバーシップを**自動生成**する。メールの実所有は検証されない。
  3. その管理者メールが**リポジトリに平文で載っている**。
- 影響: 正規管理者がまだ登録していない状態なら、**攻撃者がそのメールで先にサインアップするだけで super_admin** になり、全イベント / 全サークル / 全メンバーを掌握できる（`user.email` は unique なので「早い者勝ち」レース）。公開 SaaS では致命的。
- 推奨: (a) メール検証を必須化（`requireEmailVerification`）、(b) 初期管理者のブートストラップを「メール文字列一致で自動昇格」ではなく、ワンタイムの初期化手順 or 検証済みアカウント限定にする、(c) 管理者メールをシークレット化しリポジトリから除去、(d) 本番は `INITIAL_SUPER_ADMIN_EMAIL` を空にして手動シードに切替。

## High

### H-1. パスワード / PIN のハッシュコストが低すぎる（bcrypt saltRounds = 4）
- 場所: `circle.ts:128,133`、`membership.ts:423,682,896,1046`、その他 `bcrypt.hash(x, 4)` 全箇所。
- 問題: 「Workers の CPU 50ms 制限回避」を理由に cost=4 に固定。cost=4 は bcrypt 実用下限を大きく下回り、GPU があれば毎秒数千万〜億回試行できる。特に PIN は 4〜6 桁数字で探索空間が小さく、D1 が漏洩すればオフラインで即全解読。
- 影響: DB 流出時にサークルパスワード・管理 PIN が事実上平文同然。
- 推奨: Workers なら PBKDF2/scrypt（WebCrypto `crypto.subtle`、ネイティブ実装で高速）に移行し十分な反復回数を確保するか、`unstable_` の CPU 上限緩和 / Queue で非同期化。PIN は 4 桁固定をやめ桁数を増やす。少なくとも cost を上げられる方式へ。

### H-2. テナント分離が成立していない（無認可の横断参照）
- 場所: `circle.ts` `GET /`（13行）と `GET /:id`（48行）、`menu.ts` `GET /`（12行）・`GET /:id`（59行）、`event.ts` `GET /:id`（64行）、`membership.ts` `POST /check-permission`（342行）。
- 問題:
  - サークル一覧 / 個別、メニュー一覧 / 個別、イベント個別が**すべて無認可**。`eventId` / `circleId` を差し替えれば他テナントの店舗構成・メニュー・価格・テーマ設定を丸ごと取得できる。
  - `check-permission` は任意の `userEmail` + `circleId` で「そのメールがそのサークルで何の権限を持つか」を無認可で問い合わせでき、**メンバー構成の列挙**に使える。
- 影響: 学園祭は公開性が高いとはいえ、SaaS として複数主催者が同居する前提だと、他テナントのデータが素通し。競合テナントや無関係の第三者に内部設定が漏れる。
- 推奨: 来場者に見せてよい範囲（公開メニュー等）と、管理系（設定・メンバー）を明確に分離。管理系は認可必須化。`check-permission` は認証必須＋本人 or 管理者に限定（そもそもクライアント公開が必要か再検討）。

### H-3. リストバンド・ベアラーモデルによる個人情報の閲覧 / 改ざん・なりすまし
- 場所: `wristband.ts` `POST /onboard`（306行, 認証なし）・`lookup/:code`（81行）、`pre_order.ts` `GET /user/:code`（171行）、`order.ts` `POST /`（205行）の `userId` 任意指定、`stamp.ts` `GET /:userId`。
- 問題: `userId`（`usr_`+nanoid）や `wristbandId` を知っていれば、認証なしで
  - `onboard` で他人のニックネーム・**誕生日を上書き**、
  - `pre_order/user/:code`・`stamp/:userId` で**他人の事前注文 / スタンプ状況を閲覧**、
  - 注文作成で `userId` を他人の値にして**なりすましスタンプ付与**（→ 抽選口数・景品交換の不正）
  ができる。`displayId` は連番、`lookup` は未知コードで `eventUser` を自動生成するため、値の推測・生成が容易。
- 影響: 個人情報（誕生日）の漏洩・改ざん、景品 / 抽選の不正取得。
- 推奨: 少なくとも書き込み系（onboard・注文の `userId` 紐付け）は「リストバンド提示＝物理的近接」を担保できる導線に限定するか、来場者セッション（軽量トークン）を発行してベアラー値の推測耐性を上げる。誕生日など個人情報の収集是非自体も再検討。

### H-4. better-auth 本体（メール＋パスワード）にレート制限 / bot 対策がない
- 場所: `packages/auth/src/index.ts`。カスタム PIN / サークルパスワードには `rate-limit.ts` を入れたが、**better-auth の sign-in / sign-up には未適用**。
- 問題: `/api/auth/sign-in/email` へのオンライン総当たり、`/api/auth/sign-up/email` での大量アカウント作成（スパム / ストレージ・メール枯渇）が素通し。CAPTCHA もなし。
- 推奨: better-auth のレートリミット / verification / CAPTCHA プラグイン導入、または同等のミドルウェアを `/api/auth/*` にも適用。

### H-5. SaaS 運用に必須の周辺実装が未整備
- メール検証・**パスワードリセット / 変更**フローがない（`account.ts` にメール変更はあるがパスワード再設定導線なし）。
- **監査ログがない**（誰がいつ権限変更 / 削除したか追えない）。マルチテナントでは事故 / 不正調査に必須。
- **自動テストが皆無**（`*.test.ts` なし）。認可ロジックが手書きで各ルートに散在しているため、回帰でどこか 1 つ認可が抜けても検知できない。
- **利用規約 / プライバシーポリシー / 個人情報の取扱い**が未整備。誕生日・メール・ニックネームを収集しており、日本の個人情報保護法・退会時削除の観点で要対応（`account.ts` に削除はあるが来場者 `eventUser` の削除経路は不明瞭）。
- **バックアップ / DR**（D1 の定期バックアップ・復旧手順）が設計文書に見当たらない。

## Medium

### M-1. 無認可の自動レコード生成による DB 膨張 / コスト増 / DoS
- 場所: `wristband.ts` `lookup/:code`（144行 未知コードで `eventUser` 自動作成）・`register`（190行）、`order.ts:255`、`pre_order.ts:86`。
- 問題: 認証なしで任意コードを投げるたびに `eventUser` 行が無制限に作られる。攻撃者はスクリプトで大量生成でき、D1 の行数・課金・`displayId` 採番（`max+1` はレース）を圧迫。
- 推奨: 自動生成をやめ「存在しなければエラー」に倒す、もしくは発行導線（`/issue`, スタッフ操作）経由に限定。採番はレース安全化。

### M-2. mod（拡張機能）で任意 JS を他ユーザーのブラウザで実行できる
- 場所: `apps/register/src/components/ModSandbox.tsx`（`iframe sandbox="allow-scripts"`, `srcDoc` に `html` を直挿し、`jsUrl`/`cssUrl` に任意外部 URL）／ `circle.ts` `PATCH /:id/mods`（circle:write）。
- 評価: `allow-scripts` のみで `allow-same-origin` は付けていないため**オリジン分離は効いており**、親（管理画面）の Cookie / DOM / better-auth セッションには到達できない。ここは設計として妥当。
- 残リスク: サークル管理者が、同じサークルを見る**スタッフや来場者のブラウザ**で任意スクリプト（フィッシング UI、外部通信、マイニング等）を走らせられる。テナントをまたいだ被害ではないが、SaaS として「サークル管理者 = 半信頼」だと悪用余地がある。`jsUrl` は外部ドメイン読み込み可。
- 推奨: mod の JS 配信元を自ホスト / 許可リストに限定、CSP を iframe に付与、mod 機能をレビュー制 or オプトインにする。

### M-3. ユーザー / リソース列挙を許すエラーメッセージの差異
- 場所: `event.ts` `POST /login`（「イベントが見つかりません(404)」「サークルが見つかりません(404)」「パスワードが正しくありません(401)」を区別）。
- 問題: 応答差でイベント名・サークル名の存在有無を列挙できる。レート制限はあるが、存在確認自体が漏れる。
- 推奨: 認証失敗系は一律同一メッセージ・同一ステータスに寄せる。

### M-4. アップロードの濫用余地
- 場所: `apps/api/src/index.ts` `POST /api/upload`（148行）。
- 評価: 認証必須・拡張子制限・10MB 制限・SVG 除外は対応済みで良い。
- 残リスク: **自己登録しただけの任意ログインユーザー**（来場者含む）が 10MB ファイルを**個数無制限**に R2 へ投入できる。レート / 総量 / 所属チェックがなく、ストレージ課金・DoS の余地。
- 推奨: アップロードを「対象リソースの編集権限を持つメンバー」に限定、ユーザー単位のレート / 総容量上限を設定。

### M-5. トランザクション非対応（D1）に起因する整合性リスク
- 場所: `order.ts` 注文作成（在庫減算 → order insert → item insert を逐次実行）、`circle.ts` サークル＋代表者メンバーシップ作成、`account.ts` メール変更のカスケードなど。
- 問題: D1 が対話的トランザクション非対応のため補償なしの逐次書き込み。途中失敗で「在庫だけ減って注文が残らない」「サークルはできたが代表者不在」等の不整合が起こり得る。在庫減算はガード付き UPDATE で一部対策済みだが、後続 insert 失敗時のロールバックはない。
- 推奨: 冪等キー / 補償処理（saga）や、Durable Objects / バッチ API での原子化、最低限の整合性チェックジョブ。

### M-6. `INITIAL_SUPER_ADMIN_EMAIL` 等の設定値がリポジトリに平文
- 場所: `apps/api/wrangler.jsonc`（`vars` に本番 URL・CORS・管理者メール）。
- 問題: 管理者メールは C-1 の攻撃入力そのもの。将来他の値も `vars` に足すと秘匿情報が混入しやすい。
- 推奨: 少なくとも管理者メールはシークレット化。`vars` は非機密のみに限定する運用ルールを明文化。

## Low / 設計メモ

- **L-1 スタンプ景品の必要数がハードコード**（`stamp.ts:75` `stamps.length < 3`）。イベントごとに変えられずコメントも「仮」。SaaS ではイベント設定化が必要。
- **L-2 スキーマだけで API 未実装の機能**: `numberedTicket`（整理券）、`review`（レビュー）、`lottery*`（抽選）はテーブル定義のみで対応ルートが見当たらない。UI から使うなら未完成、使わないなら削除して攻撃面を減らす。
- **L-3 `extension.ts` が Workers 非互換**: `node:fs` / `process.cwd()` / ローカル `FesOrder-Extension-Sample` ディレクトリ依存で、本番 Worker では動作しない（デッドコード or 壊れている）。マウントされたままなら撤去 or 実装修正を。パストラバーサル対策も `..` 文字列チェックのみで脆弱。
- **L-4 在庫補充と `soldOut` の非対称**: 在庫を 0 にすると `soldOut=true` になるが（`order.ts:423`）、在庫を戻す `PATCH /:id/stock` は `soldOut` を戻さない。補充しても売切表示のままになり得る。
- **L-5 `logger()` が全リクエストを記録**: URL に `userId` / `wristbandId` / メール等が載る経路があり、ログに PII が残る。ログ保持・マスキング方針を。
- **L-6 レート制限の IP バケット共有**: `clientIp` が取れない経路では全員 `"unknown"` を共有し、正当ユーザーの巻き込みロック（可用性低下）や、逆に多数 IP からの単一対象攻撃の一部すり抜けが起こり得る。本番は CF-Connecting-IP 前提だが、経路保証を明記。
- **L-7 注文番号が推測可能**（`circleId[:4]-MMDD-連番`）。`by-number` は認可必須化済みなので実害は限定的だが、連番はビジネス情報（受注数）を露出する。

## 実装状況（2026-07-13 SaaS改善対応）

SaaS化へのフィードバックに基づき、以下のように対応済みです。

| 項目 | 状態 | 対応概要 |
|------|------|----------|
| C-1 | ✅ 対応 | `getAdminSession`／`checkMemberWritePermission` の初期管理者自動昇格を **`emailVerified===true` 限定**に。認証手段を Google/パスキーに一本化しメール/パスワード認証を廃止したため、メール検証インフラを接続せずとも検証済みメールアドレスの取得を担保（対応完了）。 |
| H-1 | ✅ 対応 | 全 `bcrypt.hash(x,4)` を **PBKDF2(WebCrypto)** の共有 util `apps/api/src/utils/password.ts` に置換。`verifySecret` が旧 bcrypt ハッシュも検証、PIN 認証/サークルログインで **rehash-on-verify** による自動移行を実装。 |
| H-2 | ✅ 対応 | circle 一覧/個別の `managerEmail`(PII) を **member:read 認可者のみに付与**（匿名来場者には返さない）。`check-permission` はセッション必須＋本人チェックに限定。event/menu の公開 GET は来場者導線で必要かつ秘匿列なしのため現状維持と判断。 |
| H-3 | ⚠️ 許容/対応 | `onboard` を **write-once 化**し、さらに「誕生日（birthday）」表現とDBカラムを「お好きな日付（favorite_date）」へ完全にリプレースして PII 収集リスクを大幅低減。同一イベント内のなりすましリスクはユーザー判断により許容（対応なし）。 |
| H-4 | ✅ 対応 | `/api/auth/*` のログインを Google OAuth 及びパスキーに絞ることで、偽メールによる大量サインアップリスクを排除。IPバケットのレート制限も追加済み。 |
| M-1 | ✅ 対応 | `wristband lookup/:code` の未知コード自動生成を撤去し 404 に。 |
| M-2 | ✅ 対応 | ModSandbox(register/visitor) の iframe `srcDoc` に制限的 CSP を付与（`connect-src 'none'` で外部送信遮断等）。 |
| M-3 | ✅ 対応 | サークルログインの失敗メッセージ 3 種を 401＋統一文言に集約（列挙対策）。 |
| M-4 | ✅ 対応 | `/api/upload` を「アクティブなメンバーシップを 1 件以上持つユーザー」に限定（自己登録来場者の濫用を遮断）。 |
| M-5 | ✅ 対応 | D1 のトランザクション機能（`db.transaction()`）を導入し、サークル作成および注文作成を完全にアトミック化。逐次書き込みによる不整合リスクを解決。 |
| M-6 | ✅ 対応 | `INITIAL_SUPER_ADMIN_EMAIL` を `vars` から除去（シークレット運用へ）。 |
| L-1 | ⚠️ 定数化のみ | スタンプ必要数を `REQUIRED_STAMP_COUNT` に定数化。イベント/サークル単位の設定化は今後の課題。 |

| L-4 | ✅ 対応 | 在庫補充(`stockQuantity>0`)時に `soldOut=false` を併せて更新。 |
| L-5 | ⚠️ 注記のみ | `logger()` に PII 混入の注意コメントを追加（マスキング方針は未実装）。 |

### 今回スコープ外（引き続き必要な作業）
- **監査ログ・自動テスト・利用規約/プライバシー・バックアップ/DR**（H-5）。
