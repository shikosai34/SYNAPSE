## 2026/07/04: SaaSマルチテナント設計 & 顧客スキャン必須化仕様

### 1. ロール階層 (RBAC) の再定義
システム全体、イベント全体、各出店（サークル）単位でテナントを分離し、以下のロール構造を定義しました：

- **システム管理者層**
  - `super_admin`: システム最高管理者。全イベント・全サークルの操作、売上データ閲覧が可能。
  - `system_manager`: システムマネージャー。システムの設定や売上データの閲覧、一部の設定変更。
  - `system_staff`: システムスタッフ。システム全体の閲覧のみ可能。
- **イベント主催者層**
  - `event_manager`: イベント管理者。イベント全体の管理、出店（サークル）の追加・削除・編集、イベントに所属するスタッフの管理が可能。
  - `event_staff`: イベントスタッフ。イベント全体の状況閲覧、出店情報の確認、来場者の照会。
- **出店・店舗スタッフ層**
  - `circle_manager`: 出店（サークル）責任者。担当出店の設定、メニュー編集、店舗所属スタッフの管理、売上ダッシュボードの閲覧。
  - `circle_staff`: 出店レジ・調理・在庫管理スタッフ。注文入力、調理ステータス変更、在庫更新。
- **一般来場者層**
  - `visitor`: 一般の来場者。事前オーダー、スタンプラリー、自身のマイQR（リストバンドID紐付け）表示。

### 2. マルチテナントにおけるコンテキスト切り替え (Workspace Switcher)
1つのユーザーアカウントが複数のスペース（システム・イベント・複数の店舗）のメンバーシップを持てるように設計しました：
- フロントエンドのヘッダー部分に **Workspace Switcher** を配置し、アクティブなスペース（店舗／イベント／システム）をリアルタイムに切り替えることができます。
- バックエンドの API 側では、ユーザーのリクエストされたスペースとアクティブなセッションを突合し、階層構造（上位ロールは下位リソースを操作可能）に基づいた動的権限検証を実行します。

### 3. 会計時のリストバンド / QRコードスキャン必須化
来場者と出店での注文を厳密に紐付け、注文履歴やスタンプラリーを機能させるため、店舗レジ（Register）からの注文作成時に顧客スキャンを必須化しました：
- レジ画面の最上部に「顧客特定スキャンパネル」を統合。
- 会計（注文の確定）を行う前に、顧客のリストバンドまたはスマートフォン画面 of QRコードの読み取り（またはID入力）を必須とし、未スキャンの状態では「注文を確定する」ボタンを無効化（ロック）します。
- 注文（`order` テーブル）に `userId` が必須として記録され、ゲスト注文時は自動的かつトランザクション安全に来場者ユーザー登録を裏で完了させます。
